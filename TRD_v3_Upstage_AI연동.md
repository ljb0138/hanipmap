# 한입지도 × Upstage AI 업그레이드 TRD

`TRD_v2_실전구축안.md`에서 짚은 가장 큰 병목은 "메뉴·가격 데이터를 계속 채워 넣게 만드는 것"과 "규칙 기반 파싱의 한계", "제보 스팸/품질 관리 공백"이었다. 이 세 가지는 공교롭게도 Upstage의 두 제품(**Document AI**, **Solar LLM**)이 정확히 겨냥하는 문제다. 이 문서는 Upstage AI를 어디에, 어떤 흐름으로 붙일지 다룬다.

---

## 1. 왜 Upstage인가

- **Document AI(OCR)**: 한글 영수증·메뉴판·간판처럼 비정형 문서를 구조화된 데이터로 바꾸는 데 특화. 지금 제보 폼의 "메뉴명:가격, 한 줄에 하나씩" 텍스트 입력은 결국 사람이 사진을 보고 손으로 타이핑하는 구조인데, 이 타이핑 과정 자체를 없앨 수 있다.
- **Solar LLM**: 한국어 성능이 좋은 국산 LLM이라, 규칙 기반 정규식으로 짜둔 자연어 파싱(`parseBudget`, `parseWalkMax`, 태그 키워드 매칭)의 한계를 넘어서는 유연한 이해가 가능하고, 크라우드소싱 콘텐츠의 품질/스팸 판별에도 쓸 수 있다.

**원칙**: AI가 자동으로 만든 데이터라도 지금 만들어둔 승인 큐(pending → approved)를 반드시 거치게 한다. AI는 사람의 타이핑 수고를 줄여주는 보조 수단이지, 검수를 생략하는 수단이 아니다.

---

## 2. 활용 시나리오 3가지

### 시나리오 A — 메뉴판 사진 한 장으로 제보 끝내기 (최우선 추천)

**현재**: 제보자가 메뉴판을 보고 `메뉴명:가격`을 한 줄씩 직접 타이핑해야 함 → 이게 제보 전환율을 깎아먹는 가장 큰 마찰.

**개선 흐름**:
```
[제보 폼] 사진 업로드
   ↓
Supabase Storage에 이미지 저장 (임시 버킷)
   ↓
서버리스 함수 /api/extract-menu.js
   ├─ Upstage Document AI로 이미지 → 텍스트/레이아웃 추출
   └─ Upstage Solar LLM으로 추출된 텍스트를
      "[{name, price}]" 형태의 JSON으로 정제
      (예: OCR이 "아메리카노 ....... 2,500" 처럼 뽑아내도
       Solar가 "아메리카노"/2500으로 깔끔하게 정리)
   ↓
프론트엔드: 메뉴 입력 textarea에 자동으로 미리 채워짐
   ↓
제보자는 "타이핑"이 아니라 "확인/수정"만 하고 제출
   ↓
기존과 동일하게 status='pending'으로 저장 → 운영자 승인
```

- 사장님 입장에서는 "사진 한 장 찍고 확인 버튼"이 끝이라, 지금 TRD_v2에서 제안한 QR코드 딥링크 제보와 결합하면 실제 사장님 참여 장벽이 크게 낮아진다.
- OCR/LLM이 잘못 읽었을 경우를 대비해, **자동 채움 결과는 항상 수정 가능한 상태로 보여주고 강제 확인 단계를 거치게** 한다 (자동 제출 금지).

### 시나리오 B — 자연어 검색을 규칙 기반 → LLM 이해로 고도화

**현재**: `parseBudget`/`parseWalkMax`는 "8천원 이하", "300m 이내" 같은 정형화된 패턴만 인식한다. "돈 없는데 배불리 먹고 싶어", "오늘 기분이 꿀꿀해서 위로되는 거 먹고 싶다" 같은 진짜 자연스러운 문장은 못 알아듣는다.

**개선 흐름**:
```
사용자 입력 자유 텍스트
   ↓
서버리스 함수 /api/interpret-query.js
   └─ Solar LLM에 "다음 문장에서 예산/거리/상황 태그를 추출해서
      JSON으로 반환해줘" 프롬프트 + 우리 태그 목록(7종) 함께 전달
   ↓
{ budget: 8000, walkMax: 5, tags: ["hangover"] } 같은 구조화 결과
   ↓
기존 matchesFilters()/runSearch() 로직에 그대로 꽂아넣기
```

- 기존 정규식 파서는 **삭제하지 않고 1차 필터로 유지**한다. 정규식이 명확히 매칭되면 그 결과를 쓰고(무료, 즉시), 애매하거나 매칭 실패 시에만 Solar 호출로 폴백하는 하이브리드 구조를 추천 (아래 4장 비용/레이턴시 참고).

### 시나리오 C — 제보 콘텐츠 품질/스팸 자동 검수

**현재**: TRD_v2에서 지적했듯 제보 경로에 사람이 보기 전 걸러내는 장치가 전혀 없다. 운영자가 pending 큐를 열어보기 전까지 스팸인지 아닌지 알 수 없다.

**개선 흐름**:
```
제보 insert 직전, 서버리스 함수에서 Solar LLM에
"이 텍스트가 실제 식당 메뉴/정보로 타당한지,
스팸/욕설/장난인지 판별해줘" 요청
   ↓
{ valid: true/false, reason: "..." } 응답
   ↓
valid=false면 애초에 저장하지 않고 제보자에게 안내
valid=true면 기존처럼 pending으로 저장 (여전히 사람 승인 필요)
```
- 이건 Rate limit/CAPTCHA(TRD_v2에서 지적한 스팸 방지책)를 **대체하는 게 아니라 보완**하는 것이다. 봇이 초당 수백 건을 찔러넣는 물리적 공격은 여전히 rate limit로 막아야 하고, LLM 검수는 "사람이 느리게 입력한 이상한 내용"을 걸러내는 역할.

## 3. 우선순위 제안

| 순서 | 시나리오 | 이유 |
|---|---|---|
| 1 | A. 메뉴판 사진 OCR | 지금 가장 큰 병목(제보 마찰)을 직접 해결. 효과가 가장 크고 즉시 체감됨 |
| 2 | C. 콘텐츠 검수 | 시나리오 A로 제보가 늘어나기 시작하면 그만큼 품질 관리 필요성도 커짐 |
| 3 | B. 자연어 검색 고도화 | 사용자 경험 개선이지만, 데이터가 충분히 쌓이기 전엔 체감 효과가 작음 (메뉴가 없는 식당이 태반이면 아무리 질의를 잘 이해해도 추천할 게 없음) |

**결론**: 데이터가 없는 게 병목인 지금 단계에서는, LLM을 검색 이해에 먼저 붙이기보다 **데이터를 채우는 입구(제보)에 먼저 붙이는 게 ROI가 높다.**

## 4. 비용/레이턴시/신뢰성 고려사항

- **비용**: Upstage API는 호출당 과금. 시나리오 B(실시간 검색)처럼 사용자 액션마다 호출되는 경로는 트래픽에 비례해 비용이 선형으로 증가한다. 시나리오 A(제보 시에만 호출)는 호출 빈도가 훨씬 낮아 비용 통제가 쉽다 → 이것도 A를 먼저 하는 이유 중 하나.
- **레이턴시**: LLM 호출은 규정식 파싱과 달리 수백ms~수 초가 걸릴 수 있다. 시나리오 B를 붙일 경우, "추천받기" 버튼 클릭 후 로딩 상태 UI가 반드시 필요 (지금은 동기적으로 즉시 렌더링되는 구조라 이 부분 설계 변경 필요).
- **신뢰성/환각(Hallucination)**: OCR+LLM이 메뉴 가격을 잘못 읽거나 지어낼 가능성이 항상 있다. **자동 추출 결과를 최종 신뢰 데이터로 취급하지 말고, 반드시 사람이 확인하는 단계(제보자 본인 확인 + 운영자 승인, 2중 확인)를 유지**해야 한다.

---

## 5. 아키텍처 반영 사항

- 새 서버리스 함수: `/api/extract-menu.js`, `/api/interpret-query.js`, `/api/moderate-submission.js` — 기존 `/api/search.js`(네이버 지역검색)와 같은 패턴으로, Secret(`UPSTAGE_API_KEY`)은 Vercel 환경변수에만 저장하고 클라이언트에는 절대 노출하지 않는다.
- 이미지 업로드가 추가되므로 **Supabase Storage 버킷** 신설 필요 (제보 이미지 임시 저장용, 승인/거절 후 일정 기간 뒤 자동 삭제하는 정책 권장 — 무한정 쌓이면 스토리지 비용 문제)
- 제보 폼(`index.html`/`app.js`)에 파일 입력(`<input type="file">`) 및 "사진으로 메뉴 채우기" 버튼 추가, 추출 결과를 기존 `subMenu` textarea에 주입하는 로직 추가

## 6. 필요한 사전 준비 (사용자님이 할 일)

1. [upstage.ai](https://www.upstage.ai) 콘솔에서 계정 생성 후 API 키 발급 (Document AI, Solar 각각 확인 필요 — 하나의 키로 통합 제공되는지, 제품별로 분리되는지는 콘솔에서 직접 확인)
2. 발급받은 키는 **저에게 공유하지 마시고 Vercel 환경변수(`UPSTAGE_API_KEY`)로 직접 등록** (네이버 검색 Secret과 동일한 보안 원칙)
3. 예상 사용량 기준으로 요금제/무료 크레딧 한도를 먼저 확인 — 시나리오 A부터 소규모로 시작해 실제 호출량을 보고 확장 여부 결정 추천

## 7. 한 줄 요약

Upstage는 세 군데 다 붙일 수 있지만, 지금 이 서비스의 병목(메뉴 데이터 부족)을 가장 직접적으로 풀어주는 건 **Document AI + Solar로 "메뉴판 사진 → 자동 구조화"를 만드는 것**이다. 검색 질의 고도화(Solar)는 데이터가 어느 정도 쌓인 다음에 붙이는 게 순서상 맞다.
