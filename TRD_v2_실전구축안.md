# 한입지도 실전 구축 TRD (프로덕션 지향)

앞서 작성한 `TRD.md`는 "지금까지 뭘 만들었나"를 기록한 문서다. 이 문서는 반대로 "제대로 서비스를 키우려면 무엇을, 왜, 어떻게 다시 짜야 하는가"를 다룬다. 지금의 바닐라 JS 1페이지 구조는 데모/MVP로는 훌륭했지만, 실사용자가 늘고 데이터가 100→1000곳으로 커지고, 제보가 실제로 들어오기 시작하면 곳곳에서 한계가 드러난다. 그 한계와 대응을 미리 짚는다.

---

## 1. 이 프로젝트의 진짜 병목이 뭔지부터

지금까지의 과정에서 이미 드러났듯, 이 서비스의 핵심 난이도는 "지도를 띄우는 것"도 "필터링 로직"도 아니다. **"메뉴·가격이라는, 아무도 공개 API로 안 주는 데이터를 어떻게 계속 신선하게 유지하느냐"**다. 프론트엔드 프레임워크를 얼마나 세련되게 짜든, 이 문제를 안 풀면 서비스는 죽는다. 그래서 이 TRD는 데이터 확보 전략을 가장 앞에 놓는다.

---

## 2. 데이터 확보 전략 (가장 중요)

### 2.1 데이터를 세 등급으로 나눠서 관리한다

| 등급 | 내용 | 확보 방법 | 현재 상태 |
|---|---|---|---|
| Tier 0 | 상호명·주소·좌표·업종 | 카카오/네이버 로컬 API 일괄 수집 | 96곳 완료 |
| Tier 1 | 영업시간, 대표가격대 | 제보 + 네이버 플레이스 등 공개 정보 수동 확인 | 거의 없음 |
| Tier 2 | 실제 메뉴·정확한 가격, 태그 | 제보 + 큐레이션 (사람 손을 반드시 거침) | 6곳만 있음 |

이 등급 구분을 **DB 스키마에 명시적으로 노출**해야 한다 (`data_tier` 컬럼 추가 추천). 지금처럼 `menu`가 비었는지 아닌지로 암묵적으로 구분하면, 나중에 "이 식당은 위치만 확인됐고 영업시간까진 확인 안 됐다" 같은 중간 상태를 표현할 수 없다.

### 2.2 제보를 "관대하게 열어두는 것"만으로는 안 채워진다

지금 만든 제보 폼은 "누군가 알아서 와서 채워주길" 기다리는 수동적 구조다. 실제로 채워지게 하려면 **유통 채널을 기술적으로 지원**해야 한다.

- **식당별 딥링크 제보 URL**: `hanipmap.vercel.app/submit?name=동네식당` 처럼 상호명이 미리 채워진 링크를 만들고, 이걸 QR코드로 뽑아 식당에 직접 제안할 수 있게 한다. 사장님이 스캔 한 번으로 자기 가게 메뉴만 입력하면 되도록 마찰을 최소화.
- **학생회/동아리 제휴용 "미션형" UX**: "우리 학과 근처 맛집 10곳 제보하면 뱃지" 같은 게이미피케이션은 UX 설계 문제지만, 이를 지원하려면 제보 테이블에 `submitted_by`뿐 아니라 **제보자 식별자(익명 세션 ID 등)** 를 남겨 집계할 수 있어야 한다. 지금 스키마엔 이게 없다.
- **사진 업로드 기반 제보**: 지금은 텍스트 폼만 있는데, 실제로는 "메뉴판 사진 찍어서 올리기"가 훨씬 마찰이 적다. Supabase Storage에 이미지를 올리고, 운영자가(또는 향후 Vision 모델이) 승인 시점에 구조화하는 2단계 흐름을 고려할 것.

### 2.3 데이터 신선도(Freshness) 관리

메뉴·가격은 시간이 지나면 틀린 정보가 된다. 지금 스키마엔 "언제 확인된 정보인지"가 없다.

- `verified_at` 컬럼 추가, 예: 6개월 지난 데이터는 카드에 "정보가 오래됐어요, 확인해주실래요?" 배지 노출 + 재확인 유도
- 폐업 신고 기능 (간단히 "폐업/이전한 것 같아요" 버튼 → 별도 플래그 테이블에 쌓고 임계치 넘으면 자동 비공개)

### 2.4 데이터 도용/오염 대응

- 경쟁 식당을 깎아내리는 악의적 제보, 장난 데이터가 들어올 수 있다. 승인 전 큐(pending)가 있다는 것 자체가 1차 방어선이지만, **제보 폼에 rate limit이 전혀 없다**는 게 지금 가장 큰 구멍이다 (아래 5장 참고).

## 3. 프론트엔드, 지금 구조로 계속 가도 되나?

### 3.1 지금(바닐라 JS)의 장점은 진짜다
- 빌드 도구 없이 즉시 배포/디버깅 가능, 러닝커브 없음, 성능도 충분히 빠름. **기능 개수가 지금 수준(한 화면, 필터+지도+폼)이면 프레임워크 없이 가는 게 오히려 합리적**이다. 무리하게 React로 갈아엎는 게 오히려 리스크.

### 3.2 그런데 이런 신호가 보이면 프레임워크 도입을 검토해야 한다
- 페이지가 여러 개로 늘어난다 (식당 상세 페이지, 마이페이지, 관리자 페이지 등) → 라우팅 필요
- `app.js` 하나에 상태(activeTag, currentList, selectedId 등)와 DOM 조작이 뒤섞여 이미 300줄이 넘었다 — 기능이 2배로 늘면 유지보수가 급격히 어려워진다
- **추천**: 지금 당장은 그대로 두되, 다음 마일스톤(식당 상세 페이지 + 관리자 페이지)에 들어갈 때 **Next.js(또는 SvelteKit) + Vercel**로 전환. 이미 Vercel을 쓰고 있어 마이그레이션 마찰이 적다.

### 3.3 지금 구조에서 당장 손봐야 할 것 (프레임워크 도입과 무관하게)
- **식당 상세 URL 부재**: 지금은 단일 페이지라 특정 식당을 링크로 공유할 수 없다. 카카오톡으로 "여기 어때?"라고 친구한테 보내는 행위 자체가 안 됨 — 이게 실사용에서 꽤 치명적이다. 최소한 `?id=42` 형태의 쿼리 파라미터로 특정 식당을 selectRestaurant 하도록 만드는 것부터 시작.
- **모바일 최적화 검증 안 됨**: 학생 대부분은 폰으로 접속한다. 현재 반응형 미디어쿼리는 있지만 실제 터치 타겟 크기, 지도 제스처(핀치줌과 페이지 스크롤 충돌) 등은 실기기 검증이 안 됐다.
- **SEO 전무**: 단일 `index.html`이라 "성균관대 맛집" 검색에 안 걸린다. 신입생 유입을 노린다면 이게 성장의 핵심 채널인데, 지금 구조로는 불가능. (Next.js 전환 시 자동 해결됨)

## 4. 백엔드/데이터베이스, Supabase로 계속 가도 되나?

### 4.1 지금 방식(직선거리 ÷ 80m)의 한계
`walk_minutes`을 좌표 간 유클리드 거리로 추정하고 있다. 실제로는 도로를 따라 걸어야 해서, 강 건너편처럼 직선거리는 가까워도 실제로는 멀리 돌아가야 하는 경우 완전히 틀린 값이 나온다.
- **개선안**: Naver Direction 5(보행자 경로) API 또는 Kakao Mobility API로 실제 도보 경로 거리를 조회. 다만 식당이 늘어날수록 호출량이 커지므로, **최초 등록 시 1회만 계산해서 DB에 캐싱**하는 구조로 설계 (매 검색마다 재호출 X).

### 4.2 거리 필터링을 DB 레벨로 옮기기
지금은 모든 식당을 클라이언트로 불러온 뒤 JS에서 필터링한다. 96곳까지는 괜찮지만, 1000곳이 되면 매번 전체를 내려받는 게 낭비다.
- **개선안**: Supabase(Postgres)에 **PostGIS 확장**을 켜고 `geography(Point)` 컬럼을 추가, `ST_DWithin`으로 "캠퍼스 반경 Xm 이내"를 DB 쿼리 단계에서 걸러서 필요한 만큼만 가져오기.

### 4.3 관리자 도구 부재
지금 승인은 Supabase Table Editor에서 사람이 손으로 status를 바꾸는 방식이다. 제보가 하루 5건만 넘어가도 이건 못 버틴다.
- **개선안**: 아주 간단한 내부용 관리자 페이지(비밀 URL + 단일 비밀번호 정도로 시작해도 충분) — pending 목록을 카드로 보여주고 승인/거절 버튼만 있는 정도. Next.js 전환 시점에 `/admin` 라우트로 얹으면 자연스럽다.

### 4.4 스팸/어뷰징 방지 (지금 가장 심각한 보안 공백)
현재 RLS는 "누구나 pending으로 insert 가능"만 막아뒀지, **얼마나 자주 insert할 수 있는지는 전혀 제한이 없다.** 즉 스크립트로 수천 건의 쓰레기 데이터를 순식간에 꽂아넣을 수 있다.
- **개선안 (우선순위 순)**:
  1. Supabase Edge Function 또는 Vercel 서버리스 함수를 제보 경로의 유일한 진입점으로 만들고, 거기에 **IP 기반 rate limit** (예: Upstash Redis + `@upstash/ratelimit`) 적용
  2. 제출 폼에 **Cloudflare Turnstile**(무료, reCAPTCHA보다 가벼움) 같은 봇 방지 위젯 추가
  3. 문자열 길이 제한, 금칙어 필터링 등 최소한의 서버사이드 검증 추가 (지금은 클라이언트 validation만 있고 서버 측 검증이 전혀 없음 — 악의적 사용자는 폼을 거치지 않고 API를 직접 호출해 어떤 값이든 넣을 수 있다)

## 5. 운영/모니터링 — "떠 있다"와 "잘 되고 있다"는 다르다

지금은 에러가 나도, 트래픽이 얼마인지도, API 쿼터를 얼마나 쓰고 있는지도 알 방법이 없다.

- **분석 도구**: Vercel Analytics(무료 티어) 또는 Plausible을 붙여서 어떤 태그가 많이 눌리는지, 어떤 식당이 자주 조회되는지 파악 → 이걸로 "다음에 어떤 식당 메뉴부터 큐레이션할지" 우선순위를 데이터 기반으로 정할 수 있다 (지금은 감으로 정하고 있음).
- **API 쿼터 소진 모니터링**: 네이버 지도/검색, 카카오 로컬 API는 모두 일일 무료 쿼터가 있다. 트래픽이 늘면 어느 날 갑자기 지도가 안 뜨는 사고가 날 수 있다. 최소한 쿼터 임계치 알림(이메일/슬랙 webhook) 정도는 필요.
- **에러 트래킹**: 지금은 `console.error`만 있고 실제 배포 환경에서 어떤 에러가 나는지 알 방법이 없다. Sentry 무료 티어 하나만 붙여도 큰 차이.

## 6. 놓치기 쉬운 법적/정책 이슈

- **개인정보**: 제보 폼의 "제보자 이름/연락처"가 혹시라도 승인된 레코드에 그대로 남아 공개 조회(SELECT) 대상이 되면 안 된다. 지금 스키마상 `submitted_by`가 `restaurants` 테이블에 같이 있어서, 승인 시 이 값이 그대로 공개 노출될 위험이 있다 — **별도 테이블로 분리하거나, 승인 시 이 컬럼을 null 처리하는 절차가 필요**.
- **콘텐츠 정책/이용약관**: 사용자가 제보한 콘텐츠(사진, 텍스트)의 이용 권한을 명확히 하는 최소한의 안내 문구가 폼에 있어야 한다 ("제보 내용은 한입지도 서비스에 게시되며, 허위 정보 제보 시 삭제될 수 있습니다" 정도).
- **상표/업체명 관련 분쟁 가능성**: 실제 상호명을 크롤링/수집해 서비스에 노출하는 것 자체는 일반적인 지도 서비스와 동일하게 통상적 범위 내지만, 특정 업체가 "우리 정보 빼달라"고 요청할 경우의 삭제 프로세스를 미리 정해두는 게 좋다.

## 7. 테스트 전략 (지금은 전무)

`bestCombo`(메뉴 조합 계산), `hoursStatus`(자정 넘는 영업시간 처리), `parseBudget`(자연어 파싱) 같은 순수 함수들은 엣지케이스가 많다 (자정 넘는 영업시간, 예산 0원, 메뉴 1개짜리 등). 지금은 브라우저에서 눈으로 확인하는 방식뿐이라 회귀 버그를 잡기 어렵다.
- **개선안**: Vitest 같은 가벼운 테스트 러너로 이 순수 함수들만이라도 단위 테스트 추가 (DOM 관련 코드는 제외하고 로직 함수만).

## 8. 단계별 로드맵 제안

| 단계 | 목표 | 핵심 작업 |
|---|---|---|
| 지금 (완료) | 데모 가능한 MVP | 지도, 필터, 96곳 위치 데이터, 제보 폼 |
| 다음 단계 | 실사용 가능한 최소 서비스 | 제보 rate limit/봇 방지, 개인정보 분리, 식당 상세 링크(쿼리파라미터) |
| 그 다음 | 데이터 볼륨 확보 | QR코드 제보 유도, 관리자 승인 페이지, verified_at 신선도 관리 |
| 성장기 | 프레임워크 전환 | Next.js 전환, PostGIS 거리 쿼리, SEO, 분석 도구 |
| 확장기 | 신뢰도/품질 향상 | 실제 도보경로 API, 콘텐츠 정책 공식화, 에러 모니터링 |

## 9. 한 줄 요약

지도를 띄우고 필터를 만드는 건 기술적으로 이미 다 끝났다. **진짜 승부는 "메뉴·가격 데이터를 지치지 않고 계속 채워 넣게 만드는 시스템(제보 마찰 최소화 + 스팸 방지 + 신선도 관리)"에 있고, 지금 구조에서 가장 급한 구멍은 제보 경로에 rate limit이 전혀 없다는 것**이다.
